## 1. 台帳の役割

計器は値だけでなく、その値が主張できる範囲と主張できない範囲を持つ。
新しい計器はこの台帳に登録し、最低1つの既知反例と比較identityを書くまで
採用しない。

実装時は同じ定義を`config/instruments.yaml`の機械可読registryとして固定し、
台帳とregistryのinstrument id/version/statusの一致をtestする。実行時の比較は
registryを読み、Markdownをparserで読まない。

statusは次の三値とする。

- `provisional`: 出力は観測証拠にできるが、自動判断のゲートに使わない。
- `calibrated_limited`: 明記した入力領域と比較identityの内部だけで判断に使える。
- `retired`: 新規出力を禁止し、既存recordの読み取りだけに使う。

## 2. `InstrumentRecord`の共通field

全recordは次を持つ。決定的計器でもfieldを省略せず、
`model_ref=deterministic:<implementation>`、`prompt_ref=none:deterministic`と明記する。

- `instrument_id`, `instrument_version`, `registry_hash`, `status`
- `measurement_id`, `subject_ref`, `value`, `unit`, `direction`
- `input_refs` とそのhash
- `model_ref`, `prompt_ref` / `prompt_hash`
- `identities`: Atlas、detector、jury roster、reader/tokenizer等の比較条件
- `confidence`: 校正とcoverageの不確定性。決定的計算であることを妥当性1.0と読まない。
- `warnings`, `evidence_refs`, `measured_at`

`compare_records(left, right)`はregistryの`comparability_keys`がすべて一致するときだけ
deltaを返す。不一致なら`comparable=false`とwarningを返し、数値差を返さない。

## 3. 初期台帳

### 3.1 `novelty.atlas_cosine` v1 — `provisional`

- **主張**: 稿と指定Atlasの最近傍chunkのcosine距離。文学的新奇性全体ではない。
- **入力**: 稿bytes、分割規則、embedder、`AtlasIdentity`。
- **出力**: `nearest_dist`、`nearest_chunk_id`、segmentごとの値と集約規則。
- **比較identity**: instrument version、Atlas identity、embedder/quantization、分割規則。
- **最終校正日**: 未校正。M4 fixtureのnear/far方向テストのみ。
- **既知反例**: L2 cell系の`novelty=1.0` / `measured_novelty=N/A`はこの値ではない。
- **既知の盲点**: 長文の3segment平均は局所模倣を薄める。Atlasの言語・年代偏りを継承する。
- **次の校正条件**: 完全な`AtlasIdentity`で再構築後、near/farとcell候補を同一計器名で混ぜない。

### 3.2 `form_fidelity.structural_features` v1 — `calibrated_limited`

- **主張**: 登録済みkindの正規表現featureが生成文に残る率。形式美全体ではない。
- **入力**: kind、source/generated bytes、detector version。
- **出力**: 0–1の保持率、source/generated feature集、適用可否。
- **比較identity**: instrument/detector version、kind、feature集。
- **最終校正日**: 2026-07-17、S-2 pilot n=40。
- **既知反例**: content cosineは骨格保持の代理にならない（r=0.18）。
- **既知の盲点**: law/RFC以外は測定不能。featureの存在は美的な機能を保証しない。
- **次の校正条件**: 新kindの最低10例と人間判定を登録するまでkindを追加しない。

### 3.3 `fixation.poetics_lexical` v1 — `calibrated_limited`

- **主張**: 隣接する詩学版の文字2-gram自己類似の高止まり。
- **入力**: 詩学版の本文列、threshold、detector version。
- **出力**: 隣接Jaccard列、平均、threshold超過bool。
- **比較identity**: detector version、threshold、tokenization規則。
- **最終校正日**: M5受入fixture。実詩学版間の本格校正は未実施。
- **既知反例**: w0004/w0005/w0007の作品横断の修辞固着を検出できない。
- **既知の盲点**: 作品、構文、修辞役割、世界型は対象外。
- **次の校正条件**: 詩学第0/1版と今後の版で語彙固着の正負fixtureを作る。

### 3.4 `fixation.house_style` v1 — `provisional`

- **主張**: 作品間での修辞装置・世界型・役割変換を含む家風の反復。
- **入力**: 作品本文、作品/節単位、標識schema、classifier model/prompt、`author_epoch`。
- **出力**: 標識率、作品間分散、役割（地の文/引用/人物/構造）、unclassified率。
- **比較identity**: classifier/prompt/schema、分割規則、author epoch、作品集manifest。
- **最終校正日**: 未校正。w0008/w0009の実走分類は校正候補。
- **既知反例**: w0004/w0005/w0007の衣裳を変えた反復、w0008の「引用への変換」、
  意味核で除外した裏方語彙に対する`backstage_world=1.0`。
- **既知の盲点**: 分類語彙そのものへのGoodhart、一作内sectionの非独立性。
- **次の校正条件**: 上記3反例をsealed fixtureとし、役割変換を消失と数えないこと。

### 3.5 `jury.disagreement_stddev` v1 — `provisional`

- **主張**: 同一packetと同一score尺度に対するvalid juror scoreの母標準偏差。
- **入力**: score、expected/valid juror slot、roster、packet/prompt hash。
- **出力**: stddev、mean、expected/valid/invalid数、invalid理由、split票率。
- **比較identity**: score尺度、jury roster/role、prompt/packet、validity規則。
- **最終校正日**: 未校正。実走の分布観測のみ。
- **既知反例**: w0009 ad hoc reportのmax-minは同じ`disagreement`名でも別量である。
- **既知の盲点**: invalid juror除外は見かけの不一致を下げる。合意増加は改善とは限らない。
- **次の校正条件**: w0005/6の改善時収束とw0007の破壊時拡大を、平均scoreと結合して再評価する。

### 3.6 `reader.mean_logprob` v1 — `provisional`

- **主張**: 指定reader modelが査読segmentへ与えた平均token logprob。perplexityと同義ではない。
- **入力**: segment bytes、reader model/tokenizer、context、promptとlogprob設定。
- **出力**: segmentごとのmean logprob、必要なら導出perplexity、欠測理由。
- **比較identity**: model revision/quantization、tokenizer、context、segment/prompt版。
- **最終校正日**: 未校正。LLM宛作品への配線testのみ。
- **既知反例**: 現在のartifact名「perplexity曲線」は実値mean logprobを表示する。
- **既知の盲点**: tokenizer/model交換で尺度が変わる。驚きと文学的効力は同義ではない。
- **次の校正条件**: 同一model/context内の既知の低/高確率fixtureと驚きの解釈を分ける。

### 3.7 `parse.reliability` v1 — `provisional`

- **主張**: 事前登録したschemaに対する有効projectionの割合と失敗様式。model能力全体ではない。
- **入力**: raw response hash/ref、schema/prompt/model、slot、事前登録retry policy。
- **出力**: attempted/completed/valid slot数、failure taxonomy、retry数、最終status。
- **比較identity**: model、prompt/schema/parser version、retry policy、batch manifest。
- **最終校正日**: 未校正。
- **既知反例**: w0009は6 call全完了でも1 parse失敗により二次score全体が欠損した。
- **既知の盲点**: 有効JSONでも意味が不正な場合を検出しない。
- **次の校正条件**: juror slotごとの即時硬化と、retry 0/1のfixtureを事前登録で検証する。

### 3.8 `run.completion` v1 — `provisional`

- **主張**: 事前登録した閉幕batchを含むrunが自然終端まで到達したか。
- **入力**: transition events、batch manifest/reservations、termination classification。
- **出力**: `complete_normal|complete_short|resource_interrupted|incomplete_other`と根拠event。
- **比較identity**: run manifest version、閉幕batch定義、停止規則。
- **最終校正日**: 未校正。
- **既知反例**: w0009のresource-stop SHELVEを美的失敗または正常な短い完走と数えてはならない。
- **既知の盲点**: terminal stateの存在だけで閉幕入力の完全性を保証しない。
- **次の校正条件**: 予約下の正常短縮run、w0009型resource stop、完走runの3 fixture。

### 3.9 `cost.reconciled_usd` v1 — `provisional`

- **主張**: call・charge event・provider明細が照合できたAPI費用。
- **入力**: linked call/charge/provider records、scope/pool/reservation。
- **出力**: amount、currency、reconciliation status、unreconciled内訳。
- **比較identity**: 同一費用包絡、provider pricing date/model、reconciliation規則。
- **最終校正日**: 未校正。Phase 3のcall/charge照合は実装済み。
- **既知反例**: provider明細が無いw0009は`unreconciled`であり、完全照合費用と主張できない。
- **既知の盲点**: timeout後の外部課金、価格改定、owner-onlyクレジットは自動scope外。
- **次の校正条件**: provider明細の利用可能な1 runで三面照合する。

## 4. 台帳から導く不変条件

1. 計器未登録または`retired`のoutputは新規生成できない。
2. `provisional`のoutputは報告できるが、自動採用・棄却・公開・Author移行を単独で決めない。
3. 比較identityが不一致なrecordにdeltaを作らない。
4. 欠測と0.0を区別する。invalid slot除外後の値にexpected/valid数を必ず付ける。
5. confidenceは自己申告、coverage、校正誤差を区別し、「計算が決定的」を「測定が正しい」にしない。
6. 評価語彙とrecordはheld-out評価後に書き込み可能な証拠であり、当該稿の再改稿promptへ戻さない。
