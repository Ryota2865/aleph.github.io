# リポジトリ全体走査 — Claude Code からの洞察

日付: 2026-07-17
署名: Claude Code（Claude Fable 5、司令塔ロール）
対象: PLAN.md・PLAN_CHANGELOG.md・PROGRESS.md 全履歴、aleph/ 全モジュール、
tests/ 全27ファイル、works/w0001〜w0007、reports/・designs/・audits/、docs/ 公開サイト
位置づけ: 設計変更ではなく、次の設計判断に供する独立走査。sol（DESIGNER_INSIGHTS_20260714）
とは意図的に別の角度から書く。solが「測定器と記録系の一貫性」を主題としたのに対し、
本稿は**「システムの重心はどこにあり、どこに移動したか」**を主題とする。

## 要旨

1. ALEPHの創設の比喩は「空きニッチの発見」だが、実測ではL2探索層は九層の中で
   最も機能しておらず、**最良の作品はすべて批評→制約実験のループから生まれている**。
   重心の移動は既成事実であり、これを正典（PLAN）が引き受けるかどうかが
   次の設計判断の本丸である。
2. このリポジトリで最も再現性の高い失敗様式は「作成済み・未接続」である。
   契約テストは関数を検証するが配線を検証しない。配線を検証する仕組みを
   契約の一部にすべきである。
3. 最大の資産はコードでも作品でもなく、**測定器を疑う文化が記録として残っている**
   ことである。同時に最大の脆弱性は、その文化の最も鋭い批評器が外部
   （Fable5チャット、7/19失効）にあることである。

## 1. 重心の移動 — ニッチ発見装置から制約実験装置へ

PLAN §0 はALEPHを「文学的な生態系の空き地を探し、そこに棲む作品を作る装置」と
定義する。だが7作品の実績を発生源で分類すると:

- **w0001〜w0003**（SHELVE）: ニッチ報告から出発した。しかしニッチの実質は薄い
  （cell系は単一クラスタへの縮退、sparse系は「空間の空隙」でなく「テキストの続き」を
  答える傾向。2026-07-11所見）。作品の質はニッチではなく著者と査読ループが担った。
- **w0004〜w0007**（全PUBLISH）: 実質的な発生源は**批評との対話**である。w0004は
  Fable5批評後のAI固有技法初配線、w0006はw0005批評への応答、w0007は
  Fable5が提案した「自己言及の出口封鎖」実験そのもの。ニッチ報告は座標の
  後付けに近い。
- 測定側も同じことを言っている: アトラスはクラスタ4個＋ノイズ85,663
  （min_cluster_sizeスパイクで「パラメータでは直らない」ことを確認済み）、
  cell系novelty=1.000定数debt残存、C-1被覆測定はn=4で実質不成立。

つまり**「空き地の地図」はまだ一度も作品の質に寄与していない**。寄与してきたのは
(a) 作品ごとの基準書導出（§6.1）、(b) 五審級査読と不一致信号、(c) 外部批評家が
設計する制約実験、(d) 詩学の注入である。これらは全て「地図なしで動く」機構である。

この事実には二通りの応答がある:

- **応答A（現行路線）**: L2は未完成なだけであり、コーパス拡張（二次コーパス・
  Gutenberg選抜）で地図に分解能を与える。designs/corpus-expansion.md の路線。
- **応答B（正典の改訂）**: ニッチ発見を「発生源」から「検証器」へ格下げする。
  すなわちニッチは書く前に探すものではなく、書いた後に「実際に空き地に着地したか」を
  測るもの（新奇性審級）とし、発生源としては制約実験（批評→禁止・強制・操作の注入）を
  正規の機構としてPLANに昇格させる。実績はすでにこちらにある。

私見では**AとBは排他ではなく、Bを先に正典化すべき**である。コーパス拡張は数週間の
施工を要し、成否も不確実（era偏りは直るが、拡張後もHDBSCANが意味的クラスタを
返す保証はない）。一方Bは、既に起きたことを記述するだけで設計の正直さが増し、
「実験manifest付きの作品」（w0007で確立した形式）を恒久機構にできる。
0.7.19の議題として設計者（Fable5）に提案する価値がある。

## 2. 「作成済み・未接続」— このリポジトリ固有の失敗様式

実例の列挙（すべてPROGRESS.mdに記録あり）:

| 部品 | 作成時期 | 接続時期 | 未接続期間の帰結 |
|---|---|---|---|
| poetics.reflect() / fixation_check() | M5 | 2026-07-17 | w0004〜w0007完成後も詩学リフレクションが一度も走らず |
| atlas.annotate_failure()（否定的地図） | M5前後 | 2026-07-17 | SHELVE 3作の失敗座標が探索に還流せず |
| 読者審級のperplexity実測 | M8 | （M8監査で発覚） | LLM宛作品の中核測定が実ランで発火せず |
| transmute() の form_fidelity ゲート | 2026-07-17 | 同日（Fable5審査で既定有効化） | 骨格喪失の贋作が距離ゲートを素通り |
| vault.py（LLM Wiki リーダー） | M1 | **未接続のまま** | PLAN §4.5の参照知識が閉ループから一度も読まれていない |
| scoutの属性ラベリング永続化 | — | 2026-07-17 | explore毎に再計算・注釈が蓄積せず |

構造的原因は明瞭で、**マイルストーン受入テストが「関数が正しい」ことを検証し、
「実ランの経路がそこを通る」ことを検証しない**からである。FakeDeps契約は
後方互換のためにgetattrガードで新フックを無音スキップする——これは正しい設計だが、
副作用として「RealDepsに配線し忘れても全テストが緑」という盲点を作る。

**提案**: 配線の棚卸しを機械化する。例えば `tests/test_wiring_inventory.py` として
「pipeline が getattr で呼ぶ全フック名」と「RealDeps が実装する属性」の差集合を
明示的にアサートする（意図的な未配線は許可リストに理由つきで載せる）。これは
solの「共通モジュール化」提言と直交し、より安価に盲点を塞ぐ。

## 3. 測定器監査の文化 — 最大の資産と、その脆弱性

このリポジトリが7月に発見した測定器の欠陥は少なくとも7件（スコアパース0.0計上、
`bool("false")`公開判定、novelty=1.000定数、form_fidelity無相関 r=0.18、
fixation_checkのスコープ齟齬、査読セグメント境界の幻の欠陥、床の錨違い）。
重要なのは件数ではなく、**全件が「作品の問題」と「測定器の問題」を分けて記録され、
修理が契約テストに固定されている**ことである。これは研究装置としての成立を意味する。

ただし脆弱性が三つある:

1. **発見の多くが外部起点**である。M8監査（Codex）、Fable5批評、sol走査が
   発見の主経路であり、内部の五審級査読は測定器自身の欠陥を検出できない
   （構造上当然——査読は測定器なので）。Fable5サブスクは7/19に失効する。
   以後の「最も鋭い批評器」をどう確保するかは設計問題として扱うべきである。
   選択肢: (a) fable-5 API呼び出しを月次批評予算として確保（作品別$3の外に
   批評専用枠）、(b) sol/codexの定期走査を恒久化、(c) Fable5の過去批評6本から
   「批評のルーブリック」を蒸留し内部審級に追加——(c)は安価だが、ルーブリック化した
   瞬間にGoodhart対象になる矛盾を含む。(a)+(b)の併用を推す。
2. **測定器の台帳がない**。どの測定器が「何を測ると主張し、いつ最後に検証されたか」は
   PROGRESSの散在記述にしかない。計器の校正記録（instrument registry）を
   1ファイル持つ価値がある（例: `designs/instruments.md`。novelty・fidelity・
   fixation・不一致度・perplexityの5行から始めれば足りる）。
3. **不一致度が次の主計器候補**である。w0005/w0006（改善時収束）とw0007
   （破壊時爆発）の対照は、平均スコアより不一致の増減の方が改稿の成否を早く
   言い当てることを示唆する。stopping への組み込みは7/17 debtに記録済みだが、
   優先度を上げてよい。計器としての筋の良さに加え、§7.1「不一致を信号として尊重」
   という正典の理念と一致するからである。

## 4. 統治・正典・記録系（簡潔に）

- **正典と現実のずれは概ね解消された**。works/深層アーカイブ公開（問3実行）、
  SHELVE分類の実配線、`checkpoint == replay(decisions.jsonl)` 契約、docs/の
  byte契約——「宣言が現実を約束しすぎる」段階は脱した。
- 残る構造debtは既知の3点: Loop.transition と pipeline._transition の二系統併存、
  終端フックの非原子性、decision追記の冪等性。いずれも記録済みで、緊急ではない。
- **budgets.yaml のcap変更がCHANGELOGに未記録**という「正典と現実のずれ」の
  再発が7/17にも確認された（0.7.17以降）。運用ルール（気づいた時点で追記）は
  弱い対策なので、`aleph status` がconfigとCHANGELOGの最新capを突き合わせて
  警告する、という機械化も選択肢（優先度低）。
- 6,068行のPython・203緑・27テストファイルという規模感は、9層の設計に対して
  過剰でも過少でもない。コード量が抑制されているのは委任（codex/pi/hermes）と
  契約先行の効果であり、この運用は維持する価値がある。

## 5. 小さな観察（優先度低・記録のみ）

- `HarnessProvider` のstdin経由本文渡し、`scrub_secrets` の全書き出し経路強制、
  vault.pyのアクセスガードなど、**秘密と権限の扱いはこの規模のプロジェクトとして
  異例に堅い**。M0期の5連続監査の投資が効いている。
- `works/w0003`（21.3万字・SHELVE）は深層アーカイブ中で最大の「読まれていない
  資産」である。七篇連作の素材としての再利用（節単位の改稿機構が入った後の
  再訪候補）は8月のバックログに値する。
- 詩学第0版は「第1版によって焼かれるために書かれた」と自己規定しているが、
  cadence=3作ごと・初回ack未承認のため、まだ一度も改訂されていない。
  w0008完成時点でack判断が必要になる——これはオーナーの人間ゲートであり、
  忘れると詩学が「権威化した緊張」（Fable5の警告した状態）に戻る。
