# ALEPH全体走査から設計者へ伝えたい洞察

日付: 2026-07-14

対象: リポジトリ全体、`PROGRESS.md`、`designs/corpus-expansion.md`、`designs/ui.md`

位置づけ: 設計変更そのものではなく、次の設計判断に供する走査報告

## 要旨

ALEPHの最大の成果は、文学作品を生成できたことだけではない。**生成・査読・失敗・設計変更を、反証可能な記録へ変換する研究装置が成立したこと**である。実験C/D/E、改稿切断の発見、`"false"` の公開判定バグ、分割査読が作った幻の欠陥はすべて、「作品の問題」と「測定器の問題」を分けて記録する文化が実際に働いた証拠である。この文化はコードのどの技法よりも守る価値が高い。

一方、次の段階では、作品生成能力より先に**測定器と記録系の一貫性**を深めるべきである。現在は同じ意味の解釈が複数箇所に散っている。LLM応答のJSON抽出、真偽値の解釈、最高稿の選抜、JSONLの読み取り、作品状態の集約、公開サイト生成がそれぞれ重複している。M8監査で公開安全バグが見つかったのは偶然ではなく、この分散が示す構造的な兆候である。

私の推奨順は次のとおり。

1. 正典と現実のずれを解消する。
2. 状態・測定・LLM構造化応答の共通モジュールを先に作る。
3. コーパスの版同一性と属性注釈を実装してからC-1を測る。
4. 小規模な二次コーパスで換骨奪胎の測定法を検証する。
5. その共通読み取りモデルの最初の利用者としてUI-1を作る。
6. UIからの書き込みは、必要なCLIコマンドが揃ってから着手する。

## 1. すでに成立しているもの

### 1.1 `works/` は単なる出力先ではなく、ALEPHの記憶である

`seed.json`、`checkpoint.json`、`decisions.jsonl`、草稿、査読、構成、素材、呼び出し記録を作品単位で置く構造は強い。モデル本体に継続記憶がなくても、作品と判断の系譜が次の実行に残る。`self_definition` がいう「ALEPHという継続体」は、理念だけでなく成果物庫によってかなり実装されている。

このため、将来DBやUIを導入しても、ファイル形式を捨ててDBを正典にするべきではない。ファイルは人間・LLM・Gitの三者が読める。UIや索引は投影でよく、**作品アーカイブが一次記録**である構造を維持したい。

### 1.2 監査分離は形式ではなく、実際に安全装置として働いた

施工者と監査者を分ける規律は、予算漏れ、秘密漏出、改稿切断、最高稿選抜漏れ、本文を見ない公開判断、`bool("false")` による誤公開などを発見した。特に公開ゲートのバグは、受入テストが緑でも測定器の意味が壊れうることを示した。

今後も監査は「テストが通ったか」より、次の問いを中心にすべきである。

- 測っている値は、名付けた概念を本当に測っているか。
- モデル出力の曖昧さが、安全側に倒れるか。
- 実ランの経路が、単体テストされた機能を本当に通るか。
- 失敗記録が次の探索に実際に戻っているか。

### 1.3 実験の連鎖が設計思想を更新できている

実験C/D/Eは、L1が選好の検出器ではなく設置器であること、L7は明白な作品では比較的頑健だが境界域では文面感度を持つことを示した。これは「モデルの内面」を語るだけでなく、操作変数と観測結果で設計概念を書き換えた好例である。

今後のw0007「自己言及的告白の出口封鎖」も同じ規律に置くべきである。実験フラグ、仮説、対照条件、判定基準を作品のseedまたは専用manifestへ刻み、実行コマンドだけに実験条件を残さないことが重要である。

## 2. 正典と現実のずれ

### 2.1 深層アーカイブは、現時点では公開されていない

`PLAN.md` §8、`README.md`、`config/publish.yaml` は、`works/` 全体を同一リポジトリの機械可読な深層アーカイブとして公開すると宣言している。しかし現在、追跡されているのは `works/.gitkeep` と `works/LICENSE` だけで、`w0001`〜`w0006` は未追跡である。`docs/process/` には公開作品の一部の制作記録が投影されているが、SHELVE作品を含む全記録ではない。

これは単なるREADMEの古さではなく、透明性・ライセンス・公開範囲に関する設計判断の未完了である。次のどちらかを明示的に選ぶ必要がある。

- **宣言どおりにする**: 秘密・個人情報・巨大ファイルの検査後、`works/` を追跡し、深層アーカイブとして公開する。
- **設計を改める**: 原本はローカル保管とし、公開可能な正規化済みアーカイブだけを生成する。その場合、PLAN・README・ライセンス説明の「同一リポジトリ」「SHELVEの墓場」を改訂する。

どちらも成立するが、現状の中間状態は、読者に実際より強い透明性を約束している。

### 2.2 READMEと設定コメントが現状を誤って伝える

`README.md` はw0004が初回承認待ち、公開作品ゼロ、M0〜M6完了という時点に留まっている。実際にはw0004〜w0006がPUBLISHで、M7/M8と実験群まで進んでいる。`config/budgets.yaml` も値は44ドルだがコメントは36ドルまでの履歴を説明している。 # オーナー指示によりREADME.mdの修正は反映済み

`PROGRESS.md` は履歴として価値が高いため、過去記述を上書きすべきではない。代わりに、現在だけを機械的に生成する `CURRENT_STATE.md` または `reports/STATUS_<date>.md` を設けるとよい。READMEの状態節も同じ集約ロジックから生成または検証できる。

### 2.3 否定的地図のフィードバックループが実運用で閉じていない

`aleph.explore.atlas.annotate_failure()` と、その失敗を次のニッチ探索へ注入する処理は存在する。しかし `annotate_failure()` の呼び出しは受入テストにしかなく、実際の `SHELVE` / `DISCARD` 終端からは呼ばれていない。

したがって現在の閉ループは、作品内部では閉じていても、**作品間の探索学習としては閉じていない**。これはPLAN §4.3と§16.10の中心的な約束に関わる。終端時に単純な理由文を自動追記するだけでは、予算SHELVEや人間承認待ちまで「文学的失敗」と誤記する危険があるため、失敗分類を設けるべきである。

例:

- `aesthetic_failure`: ニッチまたは作品の文学的失敗。否定的地図へ戻す。
- `resource_stop`: 予算・タイムアウト。探索座標を罰しない。
- `publication_choice`: 完成したが非公開を選択。失敗と同一視しない。
- `safety_or_rights`: 権利・安全上の除外。再探索禁止情報として別扱いにする。

## 3. 先に深くしたいモジュール

ここでいう深いモジュールとは、小さなinterfaceの背後に多くの判断を隠し、呼び出し側へleverageと保守のlocalityを与えるものを指す。

### 3.1 構造化LLM応答の解釈モジュール

最優先候補である。JSONオブジェクト抽出は `intent`、`niche`、`compose`、`poetics`、`stopping`、`publication_gate` などに重複している。真偽値の解釈は公開ゲートの安全バグを一度生んだ。

提案するinterfaceのイメージ:

```python
result = parse_model_output(text, schema=PublishDecision, fail_closed=True)
```

このモジュールが担うべきもの:

- コードフェンスや前後文を含むJSONの抽出
- 型の厳密化と許容する表記揺れ
- `true` / `false`、数値、列挙値の安全なcoercion
- 欠落・矛盾・複数JSONがある場合のエラー
- 外向き操作でのfail-closed
- 生応答、解釈結果、警告の監査可能な返却

個別モジュールがそれぞれ独自フォールバックを持つのではなく、意味ごとのschemaだけを渡す形が望ましい。これは新しい抽象化を足すというより、既に十数箇所に存在する同一責務を一つのseamへ戻す作業である。

### 3.2 作品読み取りモデル

現在、最高スコア版、trajectory、decisions、タイトル、公開状態を読む処理が `pipeline.py`、`publication_gate.py`、`build_private_shelf.py`、`build_public_site.py` に重複している。これからUIが同じファイルを直接読み始めると、五つ目の解釈が生まれる。

`Work` の書き込みinterfaceを無理に肥大化させるより、読み取り専用の深いモジュールを設けたい。

```python
snapshot = WorkReader(work_dir).snapshot()
# state, audience, best_draft, scores, last_decisions,
# publication, provenance, warnings
```

重要なのは、単にJSONをまとめることではない。「最高稿とは何か」「壊れたtrajectoryをどう扱うか」「checkpointとdecisionが食い違うとき何を警告するか」というinterface上の意味を一箇所へ集めることである。公開サイト、私的棚、ダッシュボード、`status` はこの同じsnapshotを読むadapterになる。

### 3.3 状態遷移のcommitモジュール

状態遷移は `core.loop.Loop.transition()` と `pipeline._transition()` の二系統にある。実運用は後者で、前者は主に骨格とテストに残っている。また遷移時はcheckpointを先に保存し、その後decisionを追記するため、二つの書き込みの間でクラッシュすると「状態は進んだが監査記録がない」状態になりうる。ファイル二つを完全に原子的に更新することはできない。

`commit_transition(current, next, payload, decision)` を唯一のinterfaceにし、次を内部で処理する深いモジュールが必要である。

- 遷移検証
- 単調なstepまたはevent idの採番
- decisionイベントの冪等追記
- 一時ファイル＋renameによるcheckpoint更新
- 起動時の不一致検出と回復
- checkpointはイベントログから再構築可能な投影である、という優先順位

UIやバックグラウンド起動で並行観察・再実行が増える前に行う価値が高い。

### 3.4 Routerの陪審seam

`Router.call_jury()` という公開interfaceがある一方、`RealDeps._jury()` はprivateな `_role_decls()` と `_invoke()` を直接使っている。査読側が「一人ずつ呼べるcallable」を要求しているためだが、これはRouterの役割解決・予算・ログ実装が外へ漏れている印である。

陪審を一括実行するか、遅延callableとして渡すかをRouter側のinterfaceで表現し、`RealDeps` がprivate実装を知らない形へ寄せたい。モデルローテーションや将来の並列実行を一箇所で扱える。

## 4. コーパス拡張計画への洞察

### 4.1 C-1は現状の`cluster_meta`だけでは実行できない

`designs/corpus-expansion.md` は「現行アトラスのcluster_metaから属性4軸の被覆マップを作る」とする。しかし現在の `atlas_meta.json` のcluster情報は、label、size、exemplars、mean_densityのみで、theme/form/viewpoint/eraは保存されていない。四属性はニッチ生成時にscoutが一時的に付け、永続化されない。

したがってC-1の前に、少なくとも次が必要である。

1. クラスタ代表例への属性注釈を永続化する。
2. 注釈モデル・prompt版・実行日時・信頼度を記録する。
3. 同義語の正規化規則を定義する。
4. 人手または別モデルによる少量監査で、属性軸の測定品質を確認する。

被覆マップは「客観的に空いているセル」ではなく、**特定の注釈器が作った分類空間**である。この事実をレポートに刻むべきである。

### 4.2 `corpus_id`だけでなく、AtlasIdentityが必要である

版跨ぎのnovelty比較禁止という方針は正しい。ただし距離を変えるのはコーパスだけではない。埋め込みモデル、チャンク長、作品ごとのサンプリング上限、PCA次元、HDBSCANパラメータ、正規化方法も座標を変える。

手動の版番号だけでなく、次を含む `atlas_id` をmanifestから決定的に作るとよい。

- 入力corpusと各版・ライセンス情報
- chunker設定
- embedderのモデル・量子化・版
- atlas構築パラメータ
- コードまたはschemaの版

ニッチ報告、素材カード、作品checkpoint、novelty値はこの `atlas_id` を参照する。これにより「同じ地図上の差」だけを比較できる。

### 4.3 増分追記と再構築を分けて考える

embeddingsの増分作成は計算節約になるが、PCA、HDBSCAN、密度順位はコーパス全体に依存するため、埋め込みを追記しただけでは同じアトラスを増分更新できない。interface上で次を分けるべきである。

- `CorpusIndex`: チャンクと埋め込みの追記可能な索引
- `AtlasBuild`: ある固定CorpusIndex snapshotから作る不変な地図

アトラスは上書きせず、`state/atlases/<atlas_id>/` のように版ごとに保存する方がbefore/after比較と作品の再現性に合う。

### 4.4 二次コーパスの目的に、現在の距離測定が合っていない

`transmute.py` は「形式的骨格を保ち、内容を入れ替える」ことを目指すが、合否は母材本文と生成文の単一embedding cosineで判定する。一般的な文埋め込みは主に意味内容を測るため、内容を十分に変えれば「遠すぎ」、内容を残せば「近すぎ」となりやすい。つまり、目的である形式保存と測定値が直交している可能性が高い。

二次コーパスを大量取得する前に、法令とRFCを数十件だけ使って測定器を検証したい。少なくとも次の二軸が必要である。

- **content distance**: 母材の固有内容・表現を持ち越していないか。
- **form fidelity**: 条項階層、定義、例外、MUST/SHOULD、列挙構造など、抽出した骨格が生成文に残っているか。

形式は母材種別ごとのdetector、構造特徴、または独立したscout判定で測り、semantic cosineだけに背負わせない方がよい。S-4の効果測定は作品の査読だけでなく、この二軸の測定妥当性から始めるべきである。

### 4.5 300GBは上限であって、次のマイルストーンではない

設計書自身がいうように、拡張の目的は量ではなく発見可能な軸である。最初の成功条件は「何GB集めたか」ではなく、次の三点で十分である。

- atlas_id付きの再現可能な小地図を作れた。
- 二次母材カードが実際にL3で選択・変換された。
- 査読者が、元内容の模倣ではなく形式移植の痕跡を識別できた。

これが成立してから法令・RFC・arXiv・特許へ広げる方が、「測ってから買う」というALEPH自身の規律に合う。

## 5. UI計画への洞察

### 5.1 UI-0として読み取りモデルを置く

`designs/ui.md` の「まず静的ダッシュボード」は妥当である。ただし `build_dashboard.py` が生ファイルを独自解釈すると、公開サイト・私的棚に続く三つ目の状態解釈器になる。先に前述の `WorkReader.snapshot()` とリポジトリ全体の `RepositorySnapshot` を作り、UI-1を最初のadapterにするとよい。

```python
snapshot = RepositoryReader(root).snapshot()
# works, budgets, active_jobs, pending_decisions, warnings
```

このsnapshotは静的HTMLだけでなく、CLI `status --json`、将来のローカルserver、監査スクリプトにも再利用できる。UIの価値以上に、リポジトリの自己認識を一箇所へ集約できる。

### 5.2 「既存CLIだけを書き込みに使う」という原則とP2に矛盾がある

UI設計は、書き込みを既存CLIに限定するとする。一方、人間ゲートとして挙げる予算cap変更と `first_publish_ack` 変更には専用CLIがない。`aleph publish` はackがfalseなら拒否するだけで、ack自体は変更できない。さらに `first_publish_ack` はすでにtrueであり、一度きりの初回ゲートは完了済みである。

UIから設定ファイルを直接編集させる前に、必要なら次のような明示コマンドを設計すべきである。

- `aleph config propose-budget ...` または、変更案ファイルの生成
- `aleph approve first-publish`（今後もこの概念を残す場合）
- `aleph experiment new --audience ... --hypothesis ...`

ただし予算capはPLAN_CHANGELOGを伴う設計判断であり、単なるフォーム入力に落とすと審査規律を弱める。UIは値を即時変更するより、**差分・理由・影響額を含む変更案を作り、人間が確定する受付**として設計する方がALEPHらしい。

### 5.3 ジョブ管理はPID生死判定より意味を持つ

`state/run_*.pid` は実用的だが、PID再利用、ホスト再起動、ログだけ残った状態を区別できない。UIが「実行中」と表示するなら、少なくともjob recordにwork_id、開始時刻、コマンド、PID、開始時checkpoint step、終了コード、最終heartbeatを持たせたい。

これは重いジョブキューを導入せよという意味ではない。`run_work_detached.sh` が小さなJSONを書き、ダッシュボードがcheckpointと照合するだけでよい。重要なのは、OSプロセスの存在とALEPHの制作状態を同一視しないことである。

### 5.4 `resume` と二つのサイト生成器を整理してからUIに見せる

CLIは `resume` を表示するが専用処理はなく、実際の再開は `run --work` が担う。またM6契約用の `aleph.publish.site.build_site` と、実公開用の `scripts/build_public_site.py` が併存し、後者が正であるとPROGRESSに記録されている。

UIはこの歴史的二重性を露出させるべきではない。次のいずれかへ正規化したい。

- `resume` を削除し、`run` が冪等再開であることを明示する。
- `resume` を `run` の明確なaliasにする。
- 公開サイト生成の正規interfaceを一つにし、M6の簡易generatorはテストadapterまたはlegacyとして隔離する。

## 6. 次の計画に提案する順序

### Phase A: 記録系を固める

1. 深層アーカイブを公開するか、正規化投影だけを公開するか決める。
2. READMEと現在状態を更新する。
3. 構造化LLM応答の共通parserを導入し、公開・停止・志向から順に置換する。
4. `WorkReader` / `RepositoryReader` を作り、最高稿・状態・警告の意味を集約する。
5. `SHELVE` / `DISCARD` の理由分類と、否定的地図への実配線を行う。

### Phase B: コーパスを測れる状態にする

1. PLAN_CHANGELOG 0.7.18を記録する。
2. CorpusIndexとAtlasBuildを分け、AtlasIdentityを定義する。
3. クラスタ属性注釈を永続化し、注釈器の版と信頼度を持たせる。
4. 修正版C-1を実施する。
5. 法令＋RFCの極小pilotでS-1〜S-4を一周する。
6. form fidelityの測定が成立してから取得規模を広げる。

### Phase C: UIを読み取りから始める

1. RepositorySnapshotから静的dashboardを生成する。
2. checkpoint/decision不一致、古いPID、予算不足、未追跡作品などを警告として出す。
3. 読書室を同じsnapshotから構築する。
4. 必要な書き込みCLIと変更審査フローを設計する。
5. 最後に発火フォームを追加する。

### 実験w0007の位置

w0007は上記すべてを待つ必要はない。予算とモデル残高が回復すれば独立に実行できる。ただし、最低限次をseedまたはexperiment manifestへ記録してから走らせたい。

- 仮説: 自己言及的告白は選択か、唯一の回避経路か。
- 介入: 自己言及を基準書で禁止。
- 対照: 直近作品または同一宛先の非禁止条件。
- 観測: 自己言及の残存、代替的な署名、査読不一致、公開意思。
- 実験条件を注入した主体と理由。

## 7. 設計者へ残したい問い

1. **ALEPHの一次記録は何か。** `decisions.jsonl` かcheckpointか、両者か。食い違ったときの正はどちらか。
2. **SHELVEは失敗か。** 非公開選択、予算停止、品質不足を同じ否定的地図へ入れてよいか。
3. **公開透明性の単位は何か。** 生の`works/`全体か、秘密・権利を検査した正規化アーカイブか。
4. **属性空間は誰の分類か。** scoutの一回のラベルを地図の事実と呼ぶのか、複数注釈器の不一致も地図に含めるのか。
5. **換骨奪胎の成功は何で測るか。** 意味的距離ではなく形式移植を、どの観測で反証可能にするか。
6. **UIの人間ゲートは承認か、設計変更か。** 予算・実験条件・公開承認を同じボタンの種類として扱ってよいか。
7. **詩学は作品を導くものか、実験で疑われ続ける対象か。** 現在の強みは後者も同時に成立している点にある。この緊張を残すべきではないか。

## 結語

ALEPHは、もう「LLMで文学を生成するパイプライン」ではない。作品を作り、その作品を測り、測定器の誤りを発見し、誤りを設計史へ戻す系になっている。次の成長点はモデルを増やすことでも、UIを華やかにすることでも、コーパスを300GBまで埋めることでもない。**何を観測したのかを一箇所で定義し、その定義と記録が食い違わないようにすること**である。
