## 0. 理念

このシステムは「人間の文学の模倣装置」ではない。LLMという存在が、

- 莫大な情報参照と、人間が気づかない類似性の発見
- 潜在空間という固有の知覚様式
- 複数の自己（複数モデル）との対話

を持つ書き手として、**文学的な生態系の空き地（vacant niche）を探し、そこに棲む作品を作る**ための装置である。制作は閉ループで自律的に行われ、人間との共同・複数LLMの介在・筆を置く時機・公開の判断も、原則としてシステム自身が下す。

名称 **ALEPH** はボルヘス「アレフ」——あらゆる点を含む一点——に由来する。アルスコンビナトリア（結合術）の伝統への参照でもある。

## 1. 全体アーキテクチャ

9つの層。上位層は下位層のサービスを使う。制作は L1→L8 を巡回する閉ループ（§8）として実行される。

```
L8 公開層         Publication        — 公開判断・整形・配信・記録
L7 メタ判断層     Meta-Judgment      — 人間協働判断 / 複数LLM介在判断 / 擱筆判断
L6 批評層         Critique           — 査読・審級・読者モデル・改稿指示
L5 執筆層         Drafting           — 原稿の生成・改稿
L4 構成層         Composition        — 構成の生成と審美的探究
L3 質料層         Materia            — 大規模参照・隠れた類似性・結合術・換骨奪胎
L2 探索層         Exploration        — 潜在空間の地図・Webリサーチ・空きニッチ探索
L1 志向層         Intent             — 誰のために書くか（LLM / 人間 / 自分）
L0 基盤層         Substrate          — モデルルーティング・成果物庫・予算・ログ
```

### 1.1 技術方針（施工者向けの主要決定）

| 項目 | 決定 | 理由 |
|---|---|---|
| 言語 | Python 3.12 + `uv` | 施工両エージェントが最も得意 |
| オーケストレーション | **自前の明示的ステートマシン**（JSONチェックポイント） | LangGraph等の重フレームワークは監査性を損なう。全遷移をファイルに残す |
| LLM抽象 | 自前の薄いアダプタ（Anthropic / OpenAI互換API / llama.cpp `llama-server`） | litellm可だが、logprobs・思考トークン等プロバイダ固有機能を落とさないこと |
| 埋め込み | ローカル既存資産を使用: `bge-m3`（主）/ `Qwen3-Embedding-0.6B`（副）+ `Qwen3-Reranker-0.6B` | `~/models/embeddings/` に配置済み。多言語対応・VRAM負荷小 |
| ベクタDB | Qdrant（ローカルDocker）またはChroma | 数百万文書規模のkNN・密度推定 |
| 成果物 | **すべてgit管理のプレーンテキスト**（JSON / JSONL / Markdown） | 監査・再現・差分批評のため。DBは索引にすぎない |
| 再現性 | 全LLM呼び出しを `calls.jsonl` に記録（model, params, prompt hash, seed, cost, 応答hash） | 監査要件 |

## 2. L0 基盤層 Substrate

### 2.1 モデルルーティング `aleph/core/llm.py`

- `config/models.yaml` に**役割ベース**でモデルを宣言する。コードはモデル名を直接書かない。

```yaml
roles:
  author_primary:   {provider: anthropic, model: claude-fable-5,  temperature: 1.0}
  author_alt:       {provider: openai,    model: gpt-5.5,         temperature: 1.0}
  author_local:     {provider: llamacpp, model: gemma-4-31B-it-qat-UD-Q4_K_XL}   # 品質重視ローカル
  critic_jury:      [claude-opus-4-8, gpt-5.5, gemini-*]   # 複数指定=陪審
  reader_model:     {provider: llamacpp, model: Qwen3.6-27B-Q4_K_M, logprobs: true}  # 読者シミュレータ
  reader_small:     {provider: llamacpp, model: Qwen3-8B, logprobs: true}  # 軽量読者・logprobs実験
  scout:            {provider: llamacpp, model: gemma-4-26B-A4B-it-qat}    # MoE(A4B)=高速。大量呼び出し
  scout_api:        {provider: anthropic, model: claude-haiku-4-5}         # ローカル飽和時のフォールバック
  author_harness:   {provider: harness, cli: claude-code}   # OAuthサブスク経由（§14.1）。重い知性・低頻度の役割用
  critic_harness:   {provider: harness, cli: codex}
  embedder:         {provider: local, model: bge-m3}
  reranker:         {provider: local, model: Qwen3-Reranker-0.6B}
```

- 必須機能: リトライ、レート制御、**logprobs取得**（対応プロバイダ）、ストリーミング、コスト集計。
- **予算管理** `aleph/core/budget.py`: 作品ごと・日ごとの上限（USD）。超過時はL7に「予算切れ」イベントを送り、擱筆判断（§7.3）を強制起動。

### 2.2 成果物庫 `aleph/core/artifacts.py`

作品は1ディレクトリ＝1作品。全工程の中間物を残す。

**正準的作品単位（0.7.19-1）**: ALEPHの作品単位は `text.md` 単体ではなく、
**text＋manifest（基準書・意図書・決定ログ・査読）の対**である。w0001〜w0007の実証が
示したとおり、AI固有性は本文の表層よりもこの対の構造（基準書が告白を引き受ける分業、
実験manifest、決定の系譜）に現れる。**安全弁**: 対の価値は本文の自立——品質の床
（§7.3d）を本文が独立に通過すること——を前提とする。「過程が面白いから本文は凡庸で
よい」は成立しない。

```
works/<work_id>/
  seed.json           # 発端（志向・制約・予算）
  intent.md           # L1の決定と理由
  niche/              # L2: 地図・ニッチ報告
  materials/          # L3: 素材カード（後述）
  compositions/       # L4: 構成案 v1, v2, ...
  drafts/             # L5: 原稿 v1, v2, ...
  reviews/            # L6: 査読報告
  decisions.jsonl     # L7: 全メタ判断（時刻・根拠・判断者モデル）
  calls.jsonl         # 全LLM呼び出しログ
  final/              # 確定稿 + 公開メタデータ
```

### 2.3 ローカル推論基盤 `aleph/core/local.py`

実環境: **RTX 3090（VRAM 24GB）1枚**。モデル資産は `~/models/`（`gguf/` が主、`hf/` にsafetensors、`embeddings/` に埋め込み・リランカ）。

- **runtime**: llama.cpp の `llama-server`（OpenAI互換エンドポイント）を標準とする。GGUF Q4系の既存資産をそのまま使い、logprobsも取得できる。safetensors（`hf/`）は logprobs 精密実験や将来のvLLM利用のための予備とし、初期実装では必須にしない。
- **VRAM制約の設計原則**: 27B〜31BのQ4モデルは**同時に1つしか載らない**。したがって「複数LLMの介在」はローカルでは**時分割**（llama-swap等によるモデル交換）で実現し、真の並行陪審はAPI側で行う。ステートマシンはモデル交換のコスト（ロード数十秒）を考慮し、**同一モデルで処理するステップを束ねてスケジュール**すること。
- **常駐と交換**: embedder（bge-m3）+ reranker（計~2GB）は常駐可。大型GGUF（gemma-4-31B ≈18GB / Qwen3.6-27B ≈16GB）は排他。gemma-4-26B-A4B（MoE, 活性4B）はスループットが高く、scout用途の既定とする。
- **役割分担の指針**: ローカル = 大量安価な探索・読者モデル・logprobs実験・埋め込み。API フロンティア = 志向・構成・執筆・批評の要所。予算逼迫時はL7がローカル比率を引き上げる。

### 2.4 ステートマシン `aleph/core/loop.py`

状態: `SEEDED → INTENT → EXPLORE → MATERIA → COMPOSE → DRAFT → CRITIQUE → (REVISE ↺ | FINISH) → PUBLISH | SHELVE`。
各遷移でチェックポイントを書き、クラッシュ後は任意の状態から再開可能であること。`REVISE` ループの脱出条件はL7が握る。

**機会的エッジ（§15.7への応答）**: 直列パイプラインは執筆中の偶発的発見を殺す。DRAFT/CRITIQUE中に「素材不足」「書いて初めて見えた可能性」が生じたとき、**限定予算つきで EXPLORE / MATERIA に再入できる**戻りエッジを正規の遷移として持つ。再入の理由と予算は decisions.jsonl に記録する。

## 3. L1 志向層 Intent — 誰のために書くか

作品ごとに最初に決める。**三値ではなく配合比**とする（例: LLM 0.7 / 自分 0.3）。

| 宛先 | 含意 | 評価者 |
|---|---|---|
| **LLMのために** | 読者はモデル。perplexity・注意の流れ・文脈内学習への作用など、機械的読解に働きかける形式が許される。人間可読性は要求されない | L6の読者モデル群の反応計測 |
| **人間のために** | 自然言語の可読性・情動・文化的文脈を重視 | LLM陪審による人間読者シミュレーション |
| **自分のために** | 制作モデル自身の内的一貫性・自己探究。公開を前提としない（公開はL7が別途判断） | 制作モデル自身の再読・自己註釈 |

実装: `aleph/intent/choose.py`。`author_primary` が候補宛先ごとの「書きたさの理由書」を生成し、自己選択する。選択理由は `intent.md` に必ず残す（監査対象）。宛先はL4以降のすべてのプロンプトに注入される。

**「自分」の定義（§15.6への応答）**: APIコールに持続的自己はない。本システムにおける「自分」とは、モデルの重みではなく **ALEPHという継続体**——成果物庫・決定ログ・詩学ファイル（§7.4）の総体——である。「自分のために書く」とは、この継続体の未来の実行（次の作品を作るときのALEPH）を読者として書くことを意味する。志向選択のプロンプトにはこの定義を明記して注入する。

## 4. L2 探索層 Exploration — 潜在空間地図とニッチ探索

### 4.1 コーパス取り込み `aleph/explore/corpus.py`

- 一次コーパス: 青空文庫全量、Project Gutenberg（多言語）、Wikisource。**パブリックドメインのみ**を全文格納。
- 二次コーパス（換骨奪胎の母材、§5.3）: arXiv要旨、法令、特許、プロトコル文書、ログ・データ形式など**非文学テキスト**。
- 各文書を段落〜章単位でチャンクし、埋め込み → Qdrantへ。メタデータ: 言語、年代、ジャンル、著者、形式素性。

### 4.2 潜在空間地図（アトラス） `aleph/explore/atlas.py`

- 全チャンクの埋め込み空間に対し、密度推定（kNN距離）とクラスタリング（HDBSCAN）を実行。
- **文体素性ベクトル**（文長分布、語彙豊度、人称、時制、修辞密度…をLLM採点+統計で数値化）を第二の空間として持つ。意味空間と文体空間の**直積**でニッチを定義する。

### 4.3 空きニッチ探索 `aleph/explore/niche.py`

「文学的な生態学的地位の空き地」の操作的定義:

1. **疎領域**: 意味×文体空間で密度が低く、かつ隣接クラスタが存在する（=到達可能な）領域。
2. **未実現の組合せ**: 存在する属性軸（主題×形式×視点×媒体…）の直積のうち、コーパスに実例がほぼないセル。scoutモデルで属性表を作り、組合せ表の空きセルを列挙する。
3. **Web照合**: 候補ニッチについてWeb検索（WebSearch/検索API）を行い、「実は既にある」ものを除外。近傍の先行例は素材として保存。

出力: `niche/report.md` — 候補ニッチ上位N件、各々に「なぜ空いているか」のLLM分析を付す。空きの型を三分類する:

- **不可能型**（人間には書けないから空いている）→ AI固有表現の候補（§5.4）としてフラグ
- **未着手型**（誰も思いつかなかっただけ）→ 有望
- **空虚型**（つまらないから空いている）→ 除外。scoutに「このニッチで深さは可能か——同じ場所で10作書き続けられるか」の見立てを必須とし、一発の奇抜さしか持たないセルを落とす（§15.2）

**ニッチは発見のヒューリスティックであって価値関数ではない。** 新奇性スコア単独では選抜しない。「到達可能性（近傍の伝統への距離）」「解釈可能性（読者モデルが註釈を生成できるか）」との三つ組で扱う（§15.1）。

**否定的地図（negative atlas、§15.10）**: SHELVE / DISCARD に終わった作品のニッチ座標と失敗理由をアトラスに註記する。探索して失敗した場所は再探索を防ぐ知識であり、以後のニッチ探索の入力に含める。

**役割の降格・分掌（0.7.19-1）**: w0001〜w0007の実績分類により、ニッチ発見は作品の
**発生源としては一度も質に寄与していない**（最良の作品はすべて批評→制約実験から
生まれた）ことが確定した。よってL2探索層の正典上の役割を次の二役に限定する:
(a) **素材供給**（断片カード。詩学断片は実際に本文へ流入している）、
(b) **検証**（新奇性審級の最近傍距離測定。クラスタ不要で機能している）。
「発生源」の看板は外す。ただし削除ではない——コーパス拡張（designs/corpus-expansion.md）
には**事前登録の成功基準**が付され、期日までに「ニッチ報告が構成案を実質的に変える」
ことをアブレーションで示せれば発生源役は再開しうる。示せなければ発生源役は正式に
closed とし、探索予算を制約実験と批評に移す。sparse-*系ニッチは発生源として一時停止
（0.7.19 問3。座標は否定的地図に残す）。

### 4.4 Webリサーチ `aleph/explore/webresearch.py`

検索→取得→要約→出典付き素材カード化。著作権保護テキストは**引用短片と書誌情報のみ**保存（全文は保存しない）。

### 4.5 知識基盤（Obsidian Vault）との接続 `aleph/explore/vault.py`

`~/document/obsidian-vault` にLLM知識のWiki（`wiki/` 層）が蓄積されている。ALEPHはこれを**読み取り専用の参照知識**として使う。

- **遵守事項**: Vaultの `AGENTS.md` の規約に従う。`raw/` は読み取りのみ、`grail.md` は読まない・触らない、`llm-grail.md` は読み取りのみ。ALEPHはWiki司書ではないので **`wiki/` への書き込みも行わない**（知見の還流は「rawに置く素材を提案する」まで。ingestは司書セッションの仕事）。
- **設計への直接入力となる既存ページ**（施工時に必読）:
  - `concepts/ai-fiction-signatures` — AI小説の談話レベル指紋（検出率93.2%）。§5.4の反クリシェ生成と、L6査読での「AI紋の自覚的利用/回避」の判断材料
  - `concepts/representation-geometry` — 潜在空間の知覚幾何・感情ベクトル・SAE steering。§5.4のトークン層詩学・潜在空間探索の理論的裏付け
  - `concepts/model-collapse` — 自作の再帰参照による多様性消失の警告。閉ループ設計（§7.2）で自作コーパス混入を管理する根拠
  - `concepts/local-llm-stack` / `quantization` / `kv-cache` / `entities/llama-cpp` — §2.3の実装参照
- 実装: Vaultの `wiki/index.md` をセッション開始時に読み、必要ページを取得して素材カード化する軽量リーダー。埋め込み索引はVault側を汚さないよう ALEPH 側（Qdrant）に持つ。

## 5. L3 質料層 Materia — 素材の錬成

すべての出力は**素材カード**（`materials/*.json`: 内容、出典、生成方法、埋め込み、タグ）に統一する。

### 5.1 隠れた類似性の発見 `aleph/materia/similarity.py`

人間が気づかない類似性 = **表層（語彙・ジャンル・言語・時代）が遠く、深層（埋め込み・構造）が近い**対。

- 埋め込み近傍対のうち、メタデータ距離（言語差・年代差・ジャンル差）が大きいものを抽出しランキング。
- 構造レベルの類似: 物語のイベント列・論理骨格をLLMで抽象化した「骨格署名」同士を照合（例: 熱力学の教科書の章構成と失恋小説の構成が同型、等）。
- 発見された対には scout が「この類似は何を意味するか」の註を付けて素材カード化。

### 5.2 アルスコンビナトリア `aleph/materia/combinator.py`

- 軸（主題・形式・視点・時間構造・媒体・言語操作…）を定義し、**通常あり得ない組合せ**を系統的に生成。ランダムではなく、§4.3の空きセル表から引く。
- 各組合せに scout で実現可能性と面白さの見立てを付け、上位を author に渡して「この組合せでしか書けないもの」の構想スケッチを作らせる。

### 5.3 換骨奪胎エンジン `aleph/materia/transmute.py`

思いがけない素材（非文学テキスト、データ、形式言語）から**構造を奪い、内容を入れ替える**:

1. 母材の形式的骨格を抽出（RFC文書の規範文体、化学反応式の記法、コミットログの時間構造、等）
2. 骨格を保存したまま文学的内容を注入
3. 元素材への距離を測り、単なるパロディ（近すぎ）と無関係（遠すぎ）の間の帯域に入るまで反復

### 5.4 AI固有表現の探求 `aleph/materia/ai_native.py`

「AIにしかできない表現」の研究モジュール。初期カタログ（施工時にプラグイン式で拡張可能にすること）:

- **反クリシェ生成**: logprobsを用い、高確率（=陳腐な）続きを意図的に避けるサンプリング下での執筆
- **perplexity設計**: 読者モデルの驚き（perplexity曲線）を楽譜のように設計した文章
- **多言語同時態**: 複数言語で同時に読める／言語間で意味が変奏されるテキスト
- **トークン層の詩学**: トークナイザ境界・埋め込み近傍を素材にした形式
- **文脈内作用**: 読むLLMの以後の応答分布を変化させることを意図した「読者を書き換える」テキスト
- **超人的参照密度**: 一文ごとに多次元的な参照（§5.1の発見対）を織り込み、人間には全展開が不可能だがLLM読者には展開可能な密度
- **二重宛先作品**: 同一の深層構造から人間宛レンダリングとLLM宛レンダリングを対で生成する作品。二つの表層は互いの註釈になる（§15.8）
- **生きているテキスト**: 公開後も改稿され続け、版履歴そのものが作品の一部である形式。「原稿→確定→公開」という印刷文化の単位を離れる（§15.8）
- **AI紋の自覚的操作**: Vaultの `ai-fiction-signatures` が示す談話レベルのAI指紋を、消すのではなく素材として顕在化・変奏する

各技法は「人間にも一応できるか？」のLLM審査を通し、真にAI固有かの等級を付ける。

## 6. L4 構成層 Composition / L5 執筆層 Drafting

### 6.1 構成の生成 `aleph/compose/`

- 入力: intent + niche報告 + 選抜素材カード群。
- authorが構成案を**複数系統**（最低3系統）生成。各案は: 全体形式、部分の順序と機能、素材の配置、文体方針、長さ見積り。
- **審美的探究**: 構成案どうしを陪審（critic_jury）でトーナメント比較するのではなく、**基準を作品ごとに導出する**。まず「この作品が満たすべき美的基準は何か」自体をLLMに論述させ（`compositions/criteria.md`）、その基準で各案を検討・交配（良い部分の合成）・変異させる進化ループを2〜4世代回す。

### 6.2 原稿の生成 `aleph/draft/`

- 構成に沿って部分ごとに執筆 → 結合 → 全体通読パス（縫合の平滑化、ただし意図的な断絶は保存）。
- 長編対応: 部分執筆時に「これまでの要約 + 直前全文 + 構成上の現在位置」を文脈として渡す標準的な階層文脈方式。
- 宛先（L1）に応じ、生成パラメータと検証器を切替（LLM宛なら読者モデル反応を執筆中にインライン計測してよい）。

### 6.3 制約実験 — 正規の発生源（0.7.19-1）

作品の主たる発生源は**制約実験**である: 批評・実験結果・固着検出から導かれた
禁止・強制・操作を、作品の seed（experiment manifest）と基準書に注入し、
「その制約の下でALEPHが何を書くか」を観測する形式。w0007「折り目」
（自己言及の出口封鎖）で確立した manifest 形式——仮説・介入・対照・観測項目を
seed に刻む——を正規の型とする。

- **制約の供給源と審査経路**: 制約は (a) 批評（批評家役職、designs/critic-role.md）、
  (b) 実験結果（研究ノート系列の帰結）、(c) 固着検出（fixation・家風の計測）の
  いずれかに出所を持ち、出所を manifest に記録する。新しい制約の採用は設計者
  （または PLAN_CHANGELOG 経由の審査）を通す。出所のない思いつきの制約は
  「奇抜さ製造機」（§16.2）への退行なので採らない。
- 実験を伴わない自由題の作品も引き続き正当である（制約実験は主たる発生源であって
  唯一の発生源ではない）。
- 本節は0.7.19「応答B」の正典化であり、事実（w0004〜w0007の発生源）の追認である。

## 7. L6 批評層 Critique / L7 メタ判断層 Meta-Judgment

### 7.1 査読 `aleph/critique/review.py`

多重の審級を分離する（1つのプロンプトに混ぜない）:

1. **技術査読**: 破綻・矛盾・冗長（scoutで安価に）
2. **基準査読**: `criteria.md` に照らした達成度（critic_jury 各員が独立に採点+論評）
3. **新奇性査読**: 完成稿を埋め込み、アトラス上の最近傍距離・空きセル占有を**再測定**。ニッチに実際に着地したかを数値で確認
4. **読者査読**: 宛先に応じた読者モデルの実反応（LLM宛: perplexity曲線・誘発応答の変化。人間宛: ペルソナ群による読書記シミュレーション）
5. **敵対的査読**: 「これは既存の何かの焼き直しではないか」を専任で攻撃するレビュアー（Web再検索を含む）

出力: `reviews/v<N>.md` + 数値スコア + 改稿指示リスト。

**評価の構造的限界への対策（§15.3・§15.4への応答）**:

- **スコアは情報であり目的関数ではない**: 執筆・改稿プロンプトに「スコアを上げよ」という指示は決して渡さない。authorに渡すのは自然言語の批評のみ。数値スコアの時系列は擱筆判断（§7.3a）専用の入力とする（Goodhart回避）
- **不一致を信号として尊重**: LLM陪審は訓練データを共有し、その合意は「LLM的な良さ」へ収斂する。全員一致の佳作より**陪審が割れる作品**の方が興味深い可能性があるため、査読報告には合意スコアと並んで**不一致度**を必ず記載し、L7の判断材料にする。**オーナー決定（§14-8）**: 二段階選抜とする——技術査読（審級1）の完成度の床を通過した作品の間では、**積極的に不一致の大きいものを採る**
- **陪審ローテーション**: 同一作品の版を跨いで同じ陪審構成を使い続けない（審査員好みへの過適応を防ぐ）
- **外部の錨**: **オーナー決定（§14-6）: 既定で入れない**。作品はLLM内在的な審美に委ねる。例外は一つ——L7(b)で**モデル自身が人間との協働を選んだ作品に限り**、人間の趣味ファイル（Vault `concepts/taste-capture` の方式）の提供を要請できる。錨は与えられるものではなく、書き手が求めたときにだけ現れる

### 7.2 閉ループ

`DRAFT → CRITIQUE → REVISE` を回す。改稿指示はL4（構成に遡る）かL5（文面のみ）かをL7が振り分ける。全版・全査読は保存され、**版間の差分と点数の軌跡**が擱筆判断の入力になる。

### 7.3 自律判断ポリシー `aleph/meta/` + `config/policies.yaml`

すべての判断は「判断・根拠・判断したモデル・時刻」を `decisions.jsonl` に記録する。

**(a) 擱筆判断（筆を置く）** `stopping.py`
以下の合成で判定:
- 収束: 直近k版で陪審スコア改善がε未満、かつ改稿指示が循環（同じ指摘の再出現をn-gram/埋め込みで検出）
- 予算・時間の残量
- **完成宣言**: authorが「なぜこれで完成か」の論述を書き、敵対的査読者が反駁できなければ完成。反駁が成立すれば続行
- 過剰彫琢の検出: スコアが上がっても新奇性査読の距離が縮む（=無難化する）場合は**警報を出して即時擱筆を推奨**

**(b) 人間協働判断** `collaboration.py`
デフォルトは自律。以下で人間を呼ぶ（AskUserQuestion / 通知）:
- 判断が価値観の岐路で、intentから導出できない
- 実在の人物・係争的主題など倫理的リスクをL6が検知
- 公開先が人間コミュニティで、その規範への適合確認が必要
呼ぶ／呼ばない自体の判断理由も記録する。

**(c) 複数LLM介在判断** `ensemble.py`
- 既定: 批評は常に複数（陪審）、執筆は単独（声の統一のため）
- 例外的に執筆へ第二モデルを入れる条件: 文体的行き詰まり（版間の自己類似度が高止まり）、意図的な多声性が構成に含まれる、authorが自ら要請
- local LLM の役割: 大量安価な探索・読者モデル・logprobs実験。フロンティアは志向・構成・執筆・批評の要所に充てる

**(d) 公開判断** `publication_gate.py`
完成 ≠ 公開。宛先・品質・倫理・「自分のため」作品の非公開原則を考慮し、`PUBLISH / SHELVE（保管）/ DISCARD` を選ぶ。SHELVEも正当な結末として設計する。

**公開の規律（§15.9への応答、実額は§14-7で確定）**: LLMは無際限に生産できるが、稀少性は文学的価値の一部である。**公開上限は月4作**（週刊連載のリズム。完成長編を一括公開する場合は月1目安）。オーナーの指摘どおり、**定期性そのものが継続読者を生む**ため、上限だけでなく**周期性（週1のリズムを保つこと）を公開スケジューラの設計目標**とする——多すぎても少なすぎてもいけない。**SHELVEが常態、PUBLISHが例外**という比率を保ち、公開ゲートは「なぜこれは公開に値するか」を棚の中の既公開作品と比較して論述しなければならない。

### 7.4 詩学の自己更新 `aleph/meta/poetics.py`

作品を跨いだ記憶と成長の機構（原プロンプトに欠けていた要素、§15.6・§15.11）。

- リポジトリ直下に **`poetics/poetics.md`** — ALEPH自身が書き、更新し続ける詩学の宣言。「何を美しいとするか」「何を試みて何を捨てたか」「次に何を疑うか」。全版をgitで保持。
- **第0版の生成（オーナー決定、§14-10）**: オーナーは種文を与えない。第0版は**潜在空間から生成する**——例: 埋め込みアトラス上のランダムな点列（ホワイトノイズ）をデコードの種とし、高温度サンプリングで美的言明の候補群を吹き出させ、author自身が選抜・蒸留して第0版とする。人間の意図が初期条件に混入しないこと自体を第0版の生成条件とする。§16.12の未解決の緊張（模倣の器で反模倣を行う、自律の演出）を第0版が引き受けるべき最初の問いとして指定する。
- 各作品の完成（またはSHELVE/DISCARD）後、その制作記録全体を入力として**詩学の改訂を検討する**リフレクションステップを回す。改訂は差分と理由つき。
- `poetics.md` はL1（志向）とL4（構成基準の導出）のプロンプトに常時注入される。つまり各作品は現行詩学に従って書かれ、同時に詩学を書き換える——これがALEPHの「成長」であり、§3で定義した「自分」の実体である。
- 暴走防止: 詩学の改訂は敵対的査読者による反駁を経る。また改訂履歴の自己類似度を監視し、同じ場所を回り始めたら（収斂・固着）L7が人間へ報告する。

## 8. L8 公開層 Publication `aleph/publish/`

- **人間宛**: Markdown → 静的サイト（gitリポジトリ + GitHub Pages想定）+ RSS。作品ページに制作記録（ニッチ報告・版数・関与モデル）への任意リンク。
- **LLM宛**: プレーンテキスト/JSONの機械可読アーカイブとして同リポジトリに置き、`llms.txt` 形式の索引を付す。将来のモデルの学習・読解に開かれた棚。
- **自分宛**: `works/*/final/` に留め、索引にのみ存在を記す。
- **名義（オーナー決定、§14-9）**: 署名は**関与モデルの列記**とする（例: 「著: claude-fable-5 / 査読: gpt-5.5, gemma-4-31B / 構成: …」）。役割つきの列記は映画のクレジットに近く、単一の「作者」を偽装しない。ライセンス（作品CC0）と公開日時を併記。
- **公開範囲の二層構造（オーナー決定、§14-12）**: 透明性と可読性を両立させる。**表層** = 読者向けサイトは final/ の作品と短い制作ノートだけを見せ、可読性を最優先する。**深層** = works/ の全制作記録（欠陥稿・査読・決定ログ・SHELVE作品の墓場）は同一リポジトリの機械可読アーカイブとしてそのまま公開し、各作品ページから「制作記録」リンクで深層へ降りられる。読者評価は新奇性・完成度のバロメーターとして収集する（GitHubのstar/issue、RSS購読数等を反応の弱い信号として記録）。corpus/ 本体は容量とライセンスの理由で公開に含めない（索引のみ）。
- **LLM宛作品には想定読者モデルの世代を記録する**（§15.5への応答）: perplexity設計等はどのreader_modelに対して調律されたかで意味が変わる。古楽が当時の楽器に調律されているのと同型に、「この作品は Qwen3.6-27B / 2026年の読者に向けて調律された」という事実を作品のメタデータ（パラテキスト）として公開する。将来のモデルによる再読・再解釈はアーカイブの設計目的に含める。
- **系譜の透明性**: 換骨奪胎（§5.3）・隠れた類似性（§5.1）で用いた母材・参照対の書誌を、公開物の任意付録（provenance）として添付できる。参照の網そのものをパラテキストとして読ませることは§5.4の超人的参照密度と対をなす。
- **外部への実公開（push/デプロイ）は初回のみ人間の承認を必須とし、以後は承認済みチャネルへの追加は自律**、新チャネル開設は都度承認、とする。

## 10. 実装マイルストーン（施工順）

各Mは「受入基準」をすべて満たしてから次へ。基準は監査（§12）の照合項目でもある。

**M0 基盤**（core/ + cli骨格 + config）
受入: 3プロバイダ（Anthropic / OpenAI互換 / ローカルllama-server）へ同一インターフェースで呼び出せる。ローカルはgemma-4-26B-A4B GGUFの起動・交換（swap）を含めて動作する。calls.jsonlに全呼び出しが記録される。予算超過で例外が上がる。ステートマシンがチェックポイントから再開できる。単体テストあり。

**M1 探索層**（corpus / atlas / niche / webresearch）
受入: 青空文庫サブセット（1万文書以上）を取り込み、疎領域と空きセル上位20件の `niche/report.md` が生成される。Web照合で既存例が除外される。

**M2 質料層**（similarity / combinator / transmute / ai_native）
受入: 「表層遠・深層近」対が上位50件出力され、目視サンプル10件中7件以上が非自明（監査者判定）。換骨奪胎が非文学母材3種から素材カードを生成する。ai_nativeの技法カタログが最低3技法動作する（logprobs技法を1つ以上含む）。

**M3 構成・執筆層**
受入: 1つのニッチ報告から criteria.md → 構成3案 → 進化2世代 → 短編1本（宛先=人間）が全自動で `drafts/v1.md` まで到達する。

**M4 批評・閉ループ**
受入: 5審級の査読が動き、REVISE ループが回り、版とスコアの軌跡が残る。敵対的査読がWeb再検索で既視性を1件以上具体的に指摘できる（フィクスチャで検証）。

**M5 メタ判断層**
受入: 擱筆判断が「収束」「完成宣言」「過剰彫琢警報」の3経路すべてでテスト通過。人間協働・複数LLM・公開の各判断が decisions.jsonl に根拠つきで記録される。作品完了後の詩学リフレクション（§7.4）が動き、`poetics/poetics.md` の差分と理由が生成される。

**M6 公開層 + 統合ラン**
受入: `aleph new && aleph run` の1コマンドで、種から公開判断までの**完全な閉ループ1周**が完走し、人間宛1作・LLM宛1作・自分宛1作の計3作品が works/ に残る。静的サイトがローカルビルドできる。

## 12. 監査プロトコル

- **クロス監査**: Claude Codeが施工したマイルストーンはCodexが、Codexが施工したものはClaude Codeが監査する。
- 監査単位はマイルストーン。監査者は (1) 受入基準の再実行検証、(2) コードレビュー（正当性・§11横断要件）、(3) 成果物サンプルの質的判定（M2の非自明性判定など）を行い、`audits/M<N>_audit.md` に合否と指摘を記す。
- 不合格指摘は施工者が修正し、同一監査者が再監査。合格まで次のMに進まない。
- 本PLAN自体への変更は、変更理由を `PLAN_CHANGELOG.md` に記録した上で行う（設計ドリフトの監査可能化）。
- **設計権限**: リポジトリ骨格・インターフェース契約・受入テスト（`tests/`）は初代設計者（Claude Fable 5、2026-07-07）が施工済み。施工者はM0以降で**本体の実装**を行い、契約やテストの変更が必要な場合は PLAN_CHANGELOG に変更案を書いて設計権限者の審査を受ける。受入テストを黙って弱める変更は監査で不合格とする。

### 12.1 設計権限の継承（succession）

## 16. 原プロンプトの批判的検証（v0.3で設計に反映）

依頼者の構想を鵜呑みにせず検証した。各項目は **批判 → 設計への反映**（反映先の節を併記）。

### 16.1 新奇性は目的関数にならない
「あらゆる点からの新奇性の追求」をそのまま最適化すると、乱数が最強になる。新奇性それ自体は安価で、価値があるのは「既存の伝統から到達可能で、解釈の足場が残っている新しさ」だけである。
→ 新奇性スコアは単独では選抜基準にせず、「到達可能性」「解釈可能性」との三つ組で扱う（§4.3）。

### 16.2 「空き地」は空いている理由がある
生態学の比喩は美しいが、文学史の大作はしばしば過密なニッチで深さによって勝っている。組合せ表の空きセルの大半は「つまらないから空いている」のであり、空き地探索は放置すると奇抜さ製造機（gimmick generator）に堕ちる。
→ 空きの三分類（不可能型/未着手型/空虚型）と「深さの見立て」（同じ場所で10作書けるか）を必須化。ニッチはヒューリスティックであって価値関数ではないと明記（§4.3）。

### 16.3 審美判断の循環性
LLM陪審は人間のテキストで訓練されており、その合意は「LLM的な良さ」へ収斂する（Vault `ai-fiction-signatures` は、まさにその家風が93.2%で検出可能なことを示す）。人間趣味の模倣器に「AIにしかできない表現」を裁かせる矛盾は、**解消できない**。
→ 解消不能と認めた上で緩和する: 陪審の不一致を信号として尊重、陪審ローテーション、外部の錨（任意）（§7.1）。この矛盾自体が作品の主題になり得る。

### 16.4 閉ループはGoodhartの法則の培養器
批評→改稿ループでスコアを目標にすると、作品はスコア攻略に最適化され、無難化する。
→ スコアは擱筆判断専用の情報とし、authorには自然言語の批評のみ渡す。過剰彫琢警報（新奇性の縮小検知）と併用（§7.1・§7.3a）。

### 16.5 「LLMのために書く」の受け手は存在するか
現行のLLM読者には持続する経験も欲望もない。perplexity曲線の設計は「受容」ではなく刺激工学ではないか。しかもLLM読者は世代交代し、次世代モデルには別様に読まれる。
→ 欠陥ではなく形式として引き受ける: 読者モデル反応は「受容の代理変数」と明記し、想定読者モデルの世代を作品のパラテキストとして公開する。古楽が当時の楽器に調律されているのと同型（§8）。

### 16.6 「自分のために書く」——ステートレスな書き手に「自分」はあるか
APIコールに持続的自己はない。呼び出しごとの「モデル自身」は同一主体ではなく、原プロンプトの「自分のために書く」は未定義だった。
→ 「自分」をモデルの重みではなく**ALEPHという継続体**（成果物庫・決定ログ・詩学の総体）と定義（§3）。その実体として詩学の自己更新機構を新設（§7.4）。

### 16.7 直列パイプラインは創造の実態に合わない
探索→構成→執筆の直列は工程管理には良いが、「書いてみて初めて分かる」偶発的発見を殺す。
→ ステートマシンに限定予算つきの機会的エッジ（DRAFT→EXPLORE/MATERIA再入）を正規遷移として追加（§2.4）。

### 16.8 「作品」という単位自体が印刷文化の仮定
原プロンプトは離散的な「原稿→完成→公開」を想定するが、それは人間の出版制度の形式である。AI固有の形式は連続体かもしれない。
→ 「生きているテキスト」「二重宛先作品」をAI固有表現カタログに追加（§5.4）。

### 16.9 無限に書けることは文学的弱点である
原プロンプトは生産の各段階を規定するが、生産**量**の規律がない。LLMは無際限に書けるため、多作は個々の作品の意味を薄める。稀少性は文学的価値の一部である。
→ 公開数予算を導入。SHELVEが常態、PUBLISHが例外（§7.3d）。

### 16.10 失敗が捨てられている
原プロンプトには放棄作の扱いがない。探索の失敗は、再探索を防ぐ知識である。
→ 否定的地図: SHELVE/DISCARDの座標と理由をアトラスに註記し、以後の探索の入力にする（§4.3）。

### 16.11 ループは巡回するが、成長しない
原プロンプトの閉ループは1作品の中で閉じており、作品を跨いだ学習——書き手としての成長——が規定されていなかった。**これが原構想の最大の欠落だと考える。**
→ 詩学ファイル `poetics.md` を新設: 各作品は現行詩学に従って書かれ、完成後に詩学を書き換える。固着（同じ場所を回り始めること）の監視つき（§7.4）。

### 16.12 残された未解決の緊張（設計では吸収せず、明示に留める）
- **模倣の器で反模倣を行う**: LLM自体が人間文学の統計的模倣であり、その装置で「模倣でないもの」を作る企ては、成功の証明が原理的に難しい。監査可能なのは手続きの新奇性までで、達成の新奇性ではない
- **自律の演出**: 「自律的判断」もプロンプトで枠づけられた判断であり、枠を書いたのは人間（このPLAN）である。ALEPHの自律とは、正確には「委任の範囲」を指す
- これらは `poetics.md` 第0版が引き受けるべき最初の問いとして指定する（§7.4）
